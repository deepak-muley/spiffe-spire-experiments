# Security Concepts in SPIFFE & SPIRE

This document elaborates on the security model, attestation, and zero-trust principles underlying SPIFFE/SPIRE.

---

## Zero Trust Security

### Traditional Security: Perimeter-Based

```
                    ┌─────────────────────────────┐
                    │     Trusted Network         │
                    │  ┌─────┐  ┌─────┐  ┌─────┐  │
  Untrusted         │  │ A   │  │ B   │  │ C   │  │         Untrusted
  Internet    ─────►│  └──┬──┘  └──┬──┘  └──┬──┘  │◄─────  Internet
                    │     │        │        │     │
                    │  "Inside = Trusted"          │
                    └─────────────────────────────┘
```

- Trust is based on **location** (inside the network)
- Once inside, services often trust each other implicitly
- Compromised internal service can access everything

### Zero Trust: Identity-Based

```
  Every request is verified based on IDENTITY, not location

  ┌─────┐                    ┌─────┐
  │ A   │ ─── "I am A" ───►  │ B   │  Validates: Is this really A?
  └─────┘    (SVID proof)    └─────┘  Checks: Does A have permission?
```

- **Never trust, always verify** - Every connection is authenticated
- **Identity-centric** - Who is making the request? (Not: Where are they?)
- **Least privilege** - Each workload gets minimal required access

SPIFFE/SPIRE provides the **identity layer** for zero trust: cryptographically strong, verifiable workload identity.

## Attestation

Attestation is the process of **proving** who a workload or node is, using evidence from trusted sources.

### Two-Phase Attestation

SPIRE performs attestation in two phases:

```
Phase 1: NODE ATTESTATION
"Is this really the node/agent we expect?"

Phase 2: WORKLOAD ATTESTATION  
"Is this process really the workload we registered?"
```

### Node Attestation

**Purpose:** Verify the identity of the machine (node) running the SPIRE Agent.

**How it works:**
1. Agent collects proof of node identity from the platform
2. Server validates the proof independently
3. On success, agent receives its own SVID (parent identity for workloads)

**Platform-specific proofs:**

| Platform | Proof Mechanism |
|----------|-----------------|
| **Kubernetes** | Service Account token (kubelet-signed) |
| **AWS EC2** | Instance Identity Document (IID) |
| **GCP** | GCE Instance Identity Token |
| **Azure** | Managed Service Identity |
| **Bare metal / No platform** | Join token (one-time use) or X.509 certificate |

**Security benefit:** Only agents running on legitimate, attested nodes can obtain identities. A rogue process cannot spoof node identity without platform compromise.

### Workload Attestation

**Purpose:** Verify the identity of the process calling the Workload API.

**How it works:**
1. Workload calls Workload API (Unix socket)
2. Agent gets process ID of caller from kernel
3. Agent queries workload attestors (e.g., kubelet) for process metadata
4. Metadata becomes **selectors** (e.g., `k8s:ns:default`, `k8s:sa:my-sa`)
5. Agent matches selectors to registration entries
6. If match found, agent returns the appropriate SVID

**Kubernetes workload attestors** typically use:
- **Namespace** (`k8s:ns`)
- **Service Account** (`k8s:sa`)
- **Pod name** (optional, for finer granularity)

**Security benefit:** Only workloads that match registered selectors receive SVIDs. A random container cannot obtain another workload's identity.

## Selectors and Registration

### Selectors

Selectors are **attributes** that uniquely identify a workload or node. They come from attestation.

**Node selectors (examples):**
- `k8s_psat:cluster:my-cluster`
- `k8s_psat:agent_ns:spire`
- `k8s_psat:agent_sa:spire-agent`

**Workload selectors (examples):**
- `k8s:ns:default`
- `k8s:sa:frontend`
- `k8s:pod-name:frontend-abc123`

### Registration Entries

A registration entry maps:
- **SPIFFE ID** (the identity to issue)
- **Parent ID** (typically the agent's SPIFFE ID)
- **Selectors** (conditions that must match)

```
Registration Entry:
  SPIFFE ID:  spiffe://example.org/ns/default/sa/frontend
  Parent:     spiffe://example.org/ns/spire/sa/spire-agent
  Selectors:  k8s:ns:default, k8s:sa:frontend
```

When a pod in `default` namespace with service account `frontend` calls the Workload API, it receives `spiffe://example.org/ns/default/sa/frontend`.

## SVIDs and Trust Bundles

### X.509-SVID

- **Certificate** - Contains SPIFFE ID in URI SAN; signed by SPIRE CA
- **Private key** - Generated by agent; never leaves the node
- **Use case:** mTLS (mutual TLS) between services

**Security properties:**
- Short TTL (e.g., 1 hour) limits exposure
- Automatic rotation
- Private key never transmitted over network

### JWT-SVID

- **JWT** - Signed by SPIRE; contains SPIFFE ID in `sub` claim
- **Use case:** API authentication, OIDC-style flows, service-to-service HTTP auth

### Trust Bundle

- Contains the **CA certificate(s)** used to sign SVIDs in the trust domain
- Workloads use it to **validate** other workloads' SVIDs
- Distributed via Workload API

---

## Trust Domain and Federation

### Trust Domain

A trust domain is a **scope of trust**. All SVIDs issued within a trust domain are signed by the same CA (or CAs in the bundle).

- `example.org` - Your organization's trust domain
- SVIDs from `spiffe://example.org/...` are trusted by other workloads in `example.org`

### Federation

When two organizations/clusters need to trust each other:

```
Trust Domain A (cluster1.com)     Trust Domain B (cluster2.com)
         │                                    │
         └────────── Federation ──────────────┘
                    (Bundle exchange)
```

- Each side publishes its **trust bundle** to the other
- Workloads in A can validate SVIDs from B and vice versa
- Enables cross-cluster, cross-org mTLS

## Security Best Practices

| Practice | Description |
|----------|-------------|
| **Short SVID TTL** | Default ~1 hour; rotate frequently to limit compromise window |
| **Least privilege registration** | Use specific selectors (namespace + SA + pod name if needed) |
| **Secure bootstrap** | Replace default bootstrap bundle in production; use proper PKI |
| **Node attestation** | Always use platform attestation (K8s SA, cloud IID) over join tokens when possible |
| **Audit logging** | Enable SPIRE telemetry for security audits |
| **Network policies** | Restrict SPIRE Server/Agent traffic; protect Workload API socket |

## Threat Model: What SPIRE Protects Against

| Threat | How SPIRE Helps |
|--------|-----------------|
| **Impersonation** | Attestation ensures only legitimate workloads get identities |
| **Stolen credentials** | Short-lived SVIDs; rotation limits usefulness of stolen certs |
| **Compromised node** | Node attestation ties agent to specific node; workload attestation limits blast radius |
| **Replay attacks** | SVIDs are time-bound; TLS prevents replay at transport layer |
| **MITM** | mTLS with SVIDs provides mutual authentication |

## What SPIRE Does NOT Solve

- **Authorization** - SPIRE provides identity; you need policy engines (OPA, Envoy RBAC, etc.) for authorization
- **Secrets management** - Use Vault, etc. for app secrets; SPIRE is for identity
- **Network encryption** - You still need TLS/mTLS; SPIRE provides the certificates
